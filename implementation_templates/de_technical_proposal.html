1. Executive Summary

Proposal Objective: Design and implement a production-grade, real-time and batch data engineering pipeline that runs on Docker (local) and AWS (production), incorporating modern best practices for observability, governance, and scalability.

Key Requirements Met

Category Requirement Solution
Extraction Schema evolution, incremental loads dlt with automatic schema detection + Redpanda for streaming
Storage High-performance OLAP with time travel DuckDB + Apache Iceberg for ACID compliance
Processing Medallion architecture (Bronze/Silver/Gold) dbt-duckdb with Polars for heavy transformations
Orchestration Asset-based lineage, testing Dagster with software-defined assets
Quality Data validation, drift detection Great Expectations + Elementary + dbt tests
Observability Monitoring, alerting, lineage OpenTelemetry + Prometheus + Grafana + Dagster UI

Strategic Advantages

Hybrid Architecture

Single codebase supports both real-time streaming and batch processing with minimal configuration changes.

Modern Stack

Best-in-breed tools: dlt, DuckDB, Polars, Dagster, Great Expectations - all Python-native and cloud-ready.

Performance

DuckDB's columnar engine + Polars' lazy evaluation = 10-100x faster than traditional Pandas/PostgreSQL.

Cost Efficiency

Serverless compute (DuckDB), efficient storage (Iceberg), minimal infrastructure overhead.

Architectural Principles

1. Data Zone (Intermediate Layer)

Principle: Raw API/Source data is first saved as parquet files in an intermediate "data_zone" before loading to bronze.

Implementation: Extraction module fetches data and saves to data_zone/{source}_{timestamp}.parquet. This provides a backup of raw data and enables reprocessing.

2. Decoupling

Principle: Ingestion, transformation, and consumption are independent layers.

Implementation: dlt writes to Bronze (raw), dbt reads from Bronze to create Silver/Gold. Kafka/Redpanda decouples producers from consumers.

3. Incremental Loading with Composite Keys

Principle: Use composite keys to uniquely identify records and enable deduplication.

Implementation: Each data source defines composite keys (e.g., lat+lon+_fetched_at for weather data). Bronze loader checks existing keys before inserting new records.

4. Idempotency

Principle: All pipeline runs produce the same output given the same input.

Implementation: dlt tracks state, dbt uses incremental models with unique keys, Dagster enables run replays.

5. Observability-First

Principle: Monitor everything - data quality, pipeline health, lineage.

Implementation: OpenTelemetry traces, Great Expectations validations, Elementary reports, Dagster asset materialization events.

6. Schema Evolution

Principle: Handle schema changes without breaking pipelines.

Implementation: dlt auto-detects schemas, Iceberg supports schema evolution, dbt on_schema_change configurations.

7. Data Quality Gates

Principle: Bad data doesn't propagate downstream.

Implementation: Great Expectations checkpoints at Bronze→Silver boundary, dbt tests at Silver→Gold.

Design Patterns Applied

  • Lambda Architecture (Hybrid): Support both real-time (Redpanda) and batch (dlt scheduled) paths merging in Bronze layer
  • Event Sourcing: Bronze layer is immutable append-only log; all transformations are reproducible
  • CQRS (Command Query Separation): Write models (Bronze ingestion) separate from read models (Gold consumption)
  • Medallion Architecture: Data Zone → Bronze (raw) → Silver (cleaned) → Gold (aggregated) for progressive refinement
  • Circuit Breaker: Pipeline failures don't cascade; Dagster retry policies prevent runaway costs
  • Two-Stage Extraction: Data Zone stores intermediate parquet files; Bronze Loader performs incremental dedup

Data Zone Composite Keys

Each data source uses composite keys to ensure uniqueness in the Bronze layer:

Data Source Composite Key Columns Description
current_weather lat, lon, _fetched_at Unique per location and fetch time
weather_forecast lat, lon, dt, _fetched_at Unique per location, forecast time, and fetch time
geocoding lat, lon, name, country, _fetched_at Unique per location and place name
reverse_geocoding lat, lon, name, country, _fetched_at Unique per location and place name

4. Data Flow Architecture

End-to-End Data Flow

Real-Time Path (Streaming)

Source → Redpanda → dlt Consumer → DuckDB Bronze → dbt (Silver/Gold) → BI

  • Source systems emit events to Redpanda topics
  • dlt consumer reads from Redpanda with offset tracking
  • Raw events written to DuckDB Bronze layer (append-only)
  • dbt incremental models process new Bronze records to Silver/Gold
  • Dagster sensors detect new data and trigger transformations

Batch Path (Scheduled)

Source API/DB → dlt Pipeline → DuckDB Bronze → dbt (Silver/Gold) → BI

  • Dagster schedules dlt pipelines (hourly/daily/custom)
  • dlt extracts with incremental state management
  • Raw data written to DuckDB Bronze layer
  • dbt runs post-ingestion via Dagster dependencies
  • Data quality gates validate before Silver/Gold promotion

Medallion Architecture Layers

Layer Purpose Technology Data Format Retention
Bronze Immutable raw data, audit trail DuckDB + Iceberg (append-only) JSON, Parquet (schema-on-read) Indefinite (compressed S3)
Silver Cleaned, validated, deduplicated dbt incremental models Parquet (typed schema) 2 years (configurable)
Gold Business-ready aggregations dbt tables/views Parquet (optimized for queries) 1 year + rollups
Mart Purpose-built datasets for teams dbt views on Gold Views (no duplication) Virtual (no storage)

Schema Evolution Strategy

Handling Schema Changes

Change Type Bronze Layer Silver Layer Gold Layer
New Column Auto-detected by dlt, added to schema dbt on_schema_change: 'append_new_columns' Manual review, add to models if needed
Deleted Column Historical data preserved dbt continues (column becomes NULL) Remove from aggregations
Type Change Stored as-is (JSON flexible) Fail dbt run, requires migration Manual intervention
Column Reorder No impact (columnar storage) No impact (SELECT by name) No impact
Key Insight: Iceberg's schema evolution features combined with dlt's auto-detection means 90% of schema changes require zero code changes. The remaining 10% (type changes) trigger alerts for manual review.

5. Component Deep Dive

5.1 Data Ingestion Layer

dlt (data load tool)

Capabilities:

  • Schema Inference: Automatically detects and evolves schemas from source data
  • Incremental Loads: Tracks state (last timestamp, cursor) to load only new/changed records
  • Error Handling: Configurable retry policies, dead-letter queues for bad records
  • Normalization: Flattens nested JSON, creates related tables automatically
  • Data Contracts: Define expected schemas and validation rules

Configuration Strategy:

  • Staging: dlt writes to Bronze layer as Parquet files in S3/MinIO
  • Chunking: Configurable batch sizes (e.g., 10k records per file) for memory efficiency
  • Deduplication: Primary key detection prevents duplicates at ingestion
  • Late Data: Watermark-based processing with configurable lateness tolerance

Supported Data Source Types

The pipeline supports extracting data from the following source types:

  • RESTful API: HTTP-based APIs with pagination, authentication (OAuth, API keys, Bearer tokens)
  • CSV: Comma-separated values files, including compressed formats (gz, zip)
  • Text: Plain text files, log files, and unstructured text data
  • Parquet: Apache Parquet columnar format for efficient analytical queries
  • Google Sheets: Google Drive spreadsheets with automatic schema detection
Example Configuration: write_disposition='merge' for SCD Type 2, primary_key='id' for deduplication, incremental=dlt.sources.incremental('updated_at') for delta loads

Redpanda (Streaming)

Use Cases:

  • Real-time clickstream data, IoT sensors, application logs
  • CDC (Change Data Capture) from databases
  • Event-driven microservices communication

Integration:

  • dlt has native Kafka/Redpanda consumer support
  • Dagster sensors listen to topic offsets to trigger downstream processing
  • Schema Registry ensures producer/consumer compatibility

5.2 Storage Layer

DuckDB

Why DuckDB over PostgreSQL/Snowflake:

  • Performance: Columnar vectorized execution → 10-100x faster for analytics
  • Portability: Embedded database, runs locally or on Lambda (no server to manage)
  • Cost: $0 licensing, minimal infrastructure (vs Snowflake $2+/hr warehouses)
  • S3-Native: Query Parquet files directly on S3 without loading into database

Configuration:

  • Memory: Set memory_limit='8GB' for predictable resource usage
  • Parallelism: threads=4 for multi-core utilization
  • Extensions: httpfs (S3), parquet (columnar), iceberg (ACID)

Apache Iceberg

Features Leveraged:

  • ACID Transactions: Prevent partial writes, ensure read consistency
  • Time Travel: Query historical snapshots for debugging/compliance
  • Schema Evolution: Add/remove columns without rewriting data
  • Partition Evolution: Change partitioning scheme without data movement
  • Hidden Partitioning: Automatic partition pruning (users don't specify partitions)
Strategic Benefit: Iceberg enables "data lakehouse" architecture - combining data lake flexibility (cheap S3 storage) with data warehouse features (ACID, time travel, performance).

5.3 Processing Layer

Polars

When to Use Polars vs dbt:

  • Polars: Complex Python logic, multi-file joins, heavy aggregations, ML preprocessing
  • dbt: SQL-native transformations, business logic that analysts understand, incremental updates

Performance Optimizations:

  • Lazy Evaluation: Build query plan, optimize, then execute (like Spark)
  • Predicate Pushdown: Filter data early in the pipeline
  • Columnar Processing: Vectorized operations on entire columns
  • Multi-threading: Automatic parallelization across CPU cores

dbt (data build tool)

Medallion Architecture in dbt:

Layer dbt Materialization Update Strategy Testing
Bronze External tables (Iceberg) Append-only via dlt Source freshness checks
Silver Incremental (merge) Process new Bronze records Unique keys, not nulls, referential integrity
Gold Table (full refresh) or View Aggregate from Silver Accepted values, relationships, custom SQL

Data Quality Strategy:

  • Generic Tests: unique, not_null, accepted_values, relationships
  • dbt-expectations: 50+ additional tests (e.g., expect_column_values_to_be_between)
  • Custom Tests: SQL queries that return failing records
  • Elementary: Anomaly detection on row counts, nulls, freshness

5.4 Orchestration Layer

Dagster

Asset-Centric Paradigm:

  • Assets vs Tasks: Dagster models data artifacts (tables, files), not operations (tasks)
  • Declarative Dependencies: Specify what data an asset needs, Dagster determines execution order
  • Materialization: Assets are "materialized" (computed/stored), Dagster tracks versions

Key Features for Our Pipeline:

  • Sensors: Trigger pipelines when new data arrives (S3 file, Redpanda offset, API change)
  • Partitions: Time-based partitions (daily/hourly) for incremental processing
  • Resources: Share connections (DuckDB, S3, Slack) across assets
  • I/O Managers: Abstract away storage (save asset to Parquet, Iceberg, or DuckDB)
  • Schedules: Cron-based or interval-based asset materialization
Example Asset Graph:
raw_orders (dlt) → cleaned_orders (dbt Silver) → daily_sales (dbt Gold) → sales_report (BI)
Dagster automatically knows: if raw_orders updates, re-run downstream assets.

5.5 Quality & Governance Layer

Great Expectations

Validation Strategy:

  • Bronze→Silver Boundary: Run GE checkpoint before dbt Silver models
  • Expectations: Column types, value ranges, uniqueness, row count thresholds
  • Profiling: Auto-generate expectations from sample data
  • Data Docs: HTML reports showing validation results

Failure Handling:

  • Critical Validations: Pipeline stops, alert to Slack, manual review required
  • Warning Validations: Log issue, continue processing, investigate later
  • Quarantine: Move bad records to separate table for inspection

Elementary

Anomaly Detection:

  • Volume Anomalies: Row count spikes/drops (e.g., daily orders suddenly 10x normal)
  • Freshness Anomalies: Data not updated in expected time window
  • Schema Anomalies: New columns, type changes, nulls in previously non-null columns
  • Drift Detection: Statistical distribution changes (e.g., average order value shifts)

Alerting:

  • Slack notifications with chart showing anomaly
  • Email digests for daily summary
  • PagerDuty integration for critical issues

5.6 Observability Layer

OpenTelemetry + Prometheus + Grafana

Metrics Collected:

  • Pipeline Metrics: Run duration, success/failure rate, records processed
  • Data Metrics: Table row counts, data freshness, quality test pass rate
  • Infrastructure Metrics: CPU/memory usage, DuckDB query performance, S3 costs

Dashboards:

  • Pipeline Health: Success rate over time, p95 run duration, error trends
  • Data Quality: Test failures by layer, anomaly detection alerts
  • Cost Monitoring: S3 storage growth, compute hours, Redpanda throughput

Dagster Lineage UI

Features:

  • Asset Graph: Visual dependency graph showing all assets and their relationships
  • Run History: Timeline of all materializations with success/failure status
  • Backfill UI: Re-run historical partitions with date range selector
  • Catalog: Searchable index of all assets with metadata (schema, owner, SLA)

6. Data Quality & Governance Framework

Comprehensive Quality Strategy

Layer 1: Ingestion Quality (Bronze)

Quality Check Implementation Action on Failure
Schema Validation dlt data contracts - enforce expected columns/types Quarantine batch, alert data engineering
Deduplication dlt primary key detection + merge write disposition Automatic dedup, log duplicates for investigation
Null Handling dlt column hints - required vs optional fields Reject batch if required field is null
Freshness dbt source freshness checks on Bronze tables Alert if data not updated in SLA window

Layer 2: Transformation Quality (Silver)

Quality Check Implementation Action on Failure
Uniqueness dbt unique test on surrogate keys Fail dbt run, prevent downstream processing
Referential Integrity dbt relationships test (foreign keys) Fail dbt run, orphaned records logged
Value Ranges dbt-expectations expect_column_values_to_be_between Warn, log outliers for review
Type Consistency Great Expectations column type expectations Fail run, requires schema migration
Volume Anomalies Elementary anomaly detection on row counts Slack alert, investigate source system

Layer 3: Business Logic Quality (Gold)

Quality Check Implementation Action on Failure
Metric Validation Custom dbt tests - e.g., revenue = quantity * price Fail run, block downstream consumption
Completeness dbt test - all expected partitions present (e.g., all dates) Warn, investigate missing data
Distribution Drift Elementary statistical tests on metrics (mean, std dev) Slack alert with charts, manual review
Aggregate Consistency dbt test - sum of parts = whole (e.g., regional = total) Fail run, logic error detected

Governance & Compliance

Data Protection Strategy

  • PII Detection: Automated scanning (dlt plugins) for SSN, credit cards, email patterns
  • Encryption: At-rest (S3 SSE-S3) and in-transit (TLS for all connections)
  • Access Control: IAM policies for S3, DuckDB role-based access, Dagster RBAC
  • Data Retention: Automatic archival to Glacier after 2 years, deletion after 7 years (configurable)
  • Audit Logging: OpenTelemetry traces capture all data access with user/timestamp

Data Lineage Tracking

  • Dagster Asset Lineage: Automatically tracks which assets produced which outputs
  • dbt Lineage: DAG visualization showing model dependencies
  • Column-Level Lineage: dbt column descriptions + dbt-metadata-syncer → data catalog
  • Impact Analysis: "If I change this Bronze table, what Gold models break?"

Handling Data Quality Issues

Good Record / Bad Record Strategy

Scenario: Batch insert of 10,000 records, 50 fail validation

Option 1: Fail-Fast (Recommended for Critical Data)

  • Entire batch rejected
  • Great Expectations checkpoint fails → stops dbt run
  • Slack alert with sample bad records
  • Data engineer fixes source data or adjusts validation rules

Option 2: Partial Insert (Recommended for Non-Critical Data)

  • 9,950 good records → Silver layer
  • 50 bad records → quarantine_silver_tablename table
  • dbt model includes UNION ALL of good records only
  • Daily job attempts to re-process quarantined records

Implementation:

-- dbt model with quarantine logic
WITH validated AS (
  SELECT *, CASE WHEN age > 0 AND age < 120 THEN 'valid' ELSE 'invalid' END AS quality_flag
  FROM bronze.users
)
SELECT * FROM validated WHERE quality_flag = 'valid' -- Silver layer
-- Invalid records go to elementary.quarantine_users

Late Arriving Data

  • Watermark-Based Processing: dbt incremental models with WHERE event_time > max_watermark - INTERVAL '3 days'
  • Late Data Window: Accept events up to 72 hours late (configurable)
  • Backfill Support: Dagster partition backfill re-processes historical windows
  • Alerting: If late data exceeds threshold (e.g., 5% of total), Slack alert for investigation

Schema Change Management

Process:
  1. dlt detects new column in source → auto-adds to Bronze Iceberg table (non-breaking)
  2. Elementary detects schema change → Slack notification
  3. Data engineer reviews: is this expected? Update dbt models if needed
  4. If type change (breaking): dlt raises exception → manual migration required
  5. Migration script: create new versioned table, backfill historical data

Release Strategy Overview: We adopt a simplified trunk-based development model with short-lived feature branches for larger changes.
Branch Strategy
Branch Type Purpose Lifetime Merge Requirements

release/* Release preparation, version bumps 1-2 weeks CI passing + staging validation
Release Flow
1. Feature Development
Developer creates feature branch from main
  • Make changes in feature/my-new-dbt-model
  • Write tests alongside code
  • Push branch and open PR against main
2. CI Pipeline Execution
Automated checks run on PR
  • Code quality: Ruff + Black checks
  • Static analysis: mypy type checking
  • Tests: pytest + dbt test (compile only)
  • Security: dependency scanning
3. Merge to Main
CI passes, approval obtained
  • Squash merge to main
  • Delete feature branch
  • Auto-deploy to staging (if enabled)
4. Production Release
Create release tag
  • Create Git tag: v1.2.3
  • GitHub Actions triggers production deployment
  • Manual approval gate for production
  • Deployment validated via health checks

Integration Code Snippet

Dagster Sensor for Jira Integration: 
from dagster import sensor, RunRequest
from jira import JIRA
import os

@sensor(job_name="create_jira_ticket")
def jira_creation_sensor(context):
    """Monitor for data quality failures and create Jira tickets."""
    
    # Get latest quality check results
    quality_results = get_elementary_results()
    
    for failure in quality_results.get_failed_checks():
        # Check if ticket already exists
        if not jira_ticket_exists(failure):
            # Create Jira ticket
            jira = JIRA(os.getenv("JIRA_SERVER"))
            issue = jira.create_issue(
                project="DATA",
                summary=f"[Data Quality] {failure.check_name} failed on {failure.table}",
                description=failure.details,
                priority={'name': 'Highest' if failure.severity == 'critical' else 'High'},
                issuetype={'name': 'Bug'},
                labels=['data-quality', 'automated']
            )
            
            # Link to Slack message
            slack_link = f"https://company.slack.com/archives/..."
            jira.add_comment(issue, f"Slack: {slack_link}")
    
    yield RunRequest(run_key=f"jira-sync-{context.cursor}")

9. Risk Analysis & Mitigation

Risk Probability Impact Mitigation Strategy
DuckDB Scalability Limits Medium High - Monitor query performance, set thresholds
- Have migration path to ClickHouse/Snowflake ready
- DuckDB handles 100GB-1TB well; migrate if exceeds
Schema Breaking Changes Medium Medium - dlt data contracts enforce expected schemas
- Iceberg schema evolution handles non-breaking changes
- Alerting + manual review for breaking changes
Streaming Backpressure Low High - Redpanda consumer lag monitoring
- Auto-scaling for dlt consumers (ECS)
- Circuit breaker: pause ingestion if downstream slow
Data Quality Degradation Medium High - Great Expectations checkpoints block bad data
- Elementary anomaly detection catches drift
- Weekly data quality review meetings
Cost Overruns (AWS) Low Medium - AWS Budgets with alerts
- S3 lifecycle policies (archive old data)
- Spot instances for non-critical workloads
Skill Gap (Polars, Dagster) Medium Low - Team training (1 week bootcamp)
- Pair programming on new tools
- Comprehensive documentation + runbooks
